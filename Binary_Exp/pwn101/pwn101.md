- Here we are going to use the `Signed Integer Overflow` vulnerability, when we are dealing with signed integers which we are in this case, we need to see how they are represented in memory.\
When we have a negative number it will be represented using `2's complement` in memory. What happens in Signed Integer Overflow let's try to understand it using an example.

```
lets say we are dealing with an 8 bit signed numbers
the max that we can represent in 8 bit is 0111 1111: 127
And now we want to add 1 to this number.
As we are dealing with 8 bit numbers we cannot represent 128 here so the result of this addition will be -128
What happened is the leftmost bit is turned on when we added 1 to the 0111 1111
```
![explanation](images/chal5_integer_overflow_explanation.png)

- So the idea is to add the max number that we can represent in 32 bit signed integer with 1, this will overflow and the result will be a negative number, and we'll get the shell.

#### Approach
The idea is that we have to use **format string vulnerability** to perform the **execution flow hijacking**. We have to hijack the execution flow to execute the **holiday** function.

There is a format specifier `%n` that allows us to write the number of bytes printed so far into the memory. Comedy is that it overwrites memory with a function which is used to print data.

![format_specifier](images/chal8_format_specifier.png)

As we want to execute the holidays function we want to write its address but the question is where to write this address so that its picked for execution.\
We saw at the starting that the `Partial Relro` is present this allows us to overwrite `.got` entries. But which address do we actually overwrite in the `.got`, this address has to be the address that comes after the vulnerability.\
We can see that there are **puts** and **printf** function only, we cannot overwrite the printf one as this is also used in the holidays function. So we are going to overwrite the puts entry in the `.got`.

Lets understand how to abuse the `%n` specifier.
- The first thing when we have format string vuln is finding out at what position from printf perspective our input resides. So doing so we take a recognizable string and then use multiple specifiers till we reach that string.\
Easy identifiable string `ABCD` we can see as `41424344`. So our input lives at 10th position from printf perspective.

![relative_pos](images/chal8_fspos_check.png)

- The first read is reading 0x12 bytes so we initally provide with these many junk values.
- The `%n` format specifier also writes an integer 4 bytes by default, that means it will overwrite only 4 bytes. So how do we overwrite the whole value in the `.got`, we do this by writing two bytes up and then the normal ones. Overwrite got entry for puts function starting address + 2, so from a specific bytes till 4 more bytes

Debugging to check if overwrite is happening or not
![debug](images/chal8_debug.png)

```
0x40123b being the address of the holidays function
1st write -> (0x40) 64
2nd write -> (0x123b) - 64 = 4603

payload = b"%64X%13$n" + b"%4603X%14$hnAAA" + p64(got_puts_address + 2) + p64(got_puts_address)

AAA is added as padding for the memory cells
we already know that our relative position is 10 from printf
%64X%13$: 10th cell
n%4603X%: 11th cell
14$hnAAA: 12th cell
p64(got_puts_address + 2): 13th cell
p64(got_puts_address): 14th cell
```

### Challenge 9
In this challenge we explore `ret2libc` attack.

- Checksec and file info on the binary

![checksec](images/chal9_checksec.png)

- Interacting with the binary

![binary_interaction](images/chal9_interact_binary.png)

- Main function of the binary, here we can observe that gets is being used which is a vulnerable function.\
With this we can overflow the stack and hijack the execution flow.

![main](images/chal9_main.png)

- Now the question is what we want to do after hijacking the execution flow, which functions address should we overwrite with the main address with.\
As there isn't any interesting function present in the binary.

- The goal of `ret2libc` attack is to leak the addresses from `.got` of standard C library(libc).\
So how can we leak the addresses from `.got` that is by jumping to `.plt`.

- When we perform `ret2libc` we call `puts` and pass in the positions of the functions in `got` which are already loaded/dynamically resolved to get their address from library

- When the binary calls the imported function lets say puts it will call to its entry in the `plt` and the plt will `invoke dynamic linker` in case the entry of the function puts in `got` has not been resolved yet

- So we are going to use the gets function to overwrite the address of main and we will jump to puts.\
As this is binary is not compiled with `PIC` the addresses for the puts and other functions in the got will not change.\
Only the value changes once the resolution is done but not the address.

- Once this is done and we have printed the libc address and got the version of libc being used, after this we will jump back to main.\
Now that we have the version of the libc we can get the offset to functions like `system` or `execve` and pass `/bin/sh` for getting shell.

**A two stage exploit**
- first leaking addresses from got
- then come back to main and then jump to system/execve

- As we are using puts which takes in a single input that is the string that it will print.\
Based on the 64 bit binary calling convention the parametes are passed using registers. The first that is used is `RDI`, so we have to look for gadgets in binary so that we can modify the `RDI` register before jumping the plt entry of puts function.

```sh
ROPgadget --binary pwn109-1644300507645.pwn109 --depth 12 > 109_gadgets.txt

# rdi instruction we are going to use
0x00000000004012a3 : pop rdi ; ret
```

- Now we call the puts function, we do it the same way binary does, that is jump to the address as the binary is not compiled with PIC.\
Similar can be done with other address of got as well.

![puts_plt](images/chal9_plt_puts.png)
![plt](images/chal9_plt_address.png)

- Now the question is how can we get the version of the libc with the addresses that we are printing in the first phase.\
When running the exploit multiple times we can see that the first 2 and last 3 nibbles of each address is the same, regardless of the randomization introduced by ASLR.\
ASLR only randomizes a portion of the address not the complete.

![address_similarity](images/chal9_addr_similarity.png)

- This is what allows us to discover the libc version the system is using. As each version of libc has its own offsets of its functions, these offsets are represented by the last 3 nibbles.\
This we can do using libc database search.

![database_search](images/chal9_libc_database_search.png)

- With the same tool we can also get the offset of other functions present in that version of libc.

![other_offsets](images/chal9_other_offsets.png)

```

str_bin_sh 0x130c4d
```

print("Leaked puts address : {}".format(str(hex(leaked_puts_addr))))
print("Leaked gets address : {}".format(str(hex(leaked_gets_addr))))
print("Leaked setvbuf address : {}".format(str(hex(leaked_setvbuf_addr))))

# libc search reveals the version of libc being unsigned
# second stage, going back to main and spawing shell when vulnerable gets is executed

Relative to leaked_gets_address

- Now we have to check if we have the required gadgets.\
A gadget is a small snippet of assembly code already present in a program’s binary or linked libraries that ends with a ret (return) instruction.\
By overwriting return addresses with the memory addresses of these gadgets, we create a "ROP chain" where the ret instruction at the end of one gadget jumps to the next, enabling complex arbitrary code execution.\
We can look for these gadgets using `ROPgadget --binary pwn110-1644300525386.pwn110 --depth 12 > gadgets.txt`.

![rop_gadget](images/chal10_gadgets.png)

- We also need gadgets which we can use for modifying `rsi` and `rdx`, for executing `__mprotect`.

```
0x000000000040f4de : pop rsi ; ret
0x000000000040181f : pop rdx ; ret
0x0000000000463c43 : jmp rsp
0x000000000040101a : ret
```

#### Approach
- We hijack the execution flow using the `gets` function persent in main.
- With the hijack we'll make the execution go through the `puts` function which will leak the value of `__libc_stack_end`.
- Once we are able to leak the stack end address, we need to find the begining address of page memory that contains the address we leaked.
```
For example end of stack:
0x7ffc4ee6fc28
we know page size is: 0x1000
--------------
0x7FFC4EE6F000

this is done using bitwise AND between the stack address and negating (page size - 1)
```
- After this we look for gadgets for the `__mprotect` call.
- Once we have all this we can place a shellcode in the stack, but also we need to make sure we can execute it, so redirect the rip to the shellcode.\
For this too we'll look for gadgets `jmp rsp`
- What happens here is we increment the `jmp rsp` instruction to the part where we have shellcode.\
When the ret of the mprotect function happens the rip will point to the `jmp rsp` which we have already incremented to our shellcode, and hence our shellcode gets executed next.
